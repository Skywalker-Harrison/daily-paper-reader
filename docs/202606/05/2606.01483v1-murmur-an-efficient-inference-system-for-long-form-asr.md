## 摘要
长语音自动识别（ASR）既需要高精度又需要低延迟，但现有系统迫使两者之间进行权衡。基于块的流水线并行处理音频窗口以实现低延迟，但丢失了跨块上下文，并且需要脆弱的启发式方法来对齐边界处的说话人和时间戳。长上下文ASR模型通过单次处理解决所有问题，从而获得更好的准确性，但速度慢了一个数量级。我们提出了Murmur，这是一个通过两个层次运作来克服这种权衡的推理系统。在块间层次，我们重新审视基于块的流水线以适应现代长上下文ASR，将块大小视为可调超参数，并表明中间块大小在准确性和延迟之间取得了良好的平衡。在块内层次，我们通过将滑动窗口KV缓存逐出策略应用于输出和语音标记，利用注意力稀疏性。在AMI-IHM上，Murmur匹配了单次处理的准确性，同时将延迟降低了4.2倍，并且通过标记逐出进一步获得收益，相对tcpWER退化低于1%。Murmur的代码可在https://github.com/uw-syfi/Murmur获取。

## Abstract
Long-form automatic speech recognition (ASR) requires both high accuracy and low latency, but existing systems force a trade-off between the two. Chunk-based pipelines process audio in parallel windows for low latency, but lose cross-chunk context and need brittle heuristics to align speakers and timestamps at boundaries. Long-context ASR models resolve everything in a single pass for better accuracy, but are an order of magnitude slower. We propose Murmur, an inference system that overcomes this trade-off by operating at two levels. At the inter-chunk level, we revisit the chunk-based pipeline for modern long-context ASR, treating chunk size as a tunable hyperparameter, and show that intermediate chunk sizes strike a good balance of accuracy and latency. At the intra-chunk level, we exploit attention sparsity through a sliding window KV cache eviction policy applied to both output and speech tokens. On AMI-IHM, Murmur matches single-pass accuracy while reducing latency by 4.2x, with further gains from token eviction at less than 1% relative tcpWER degradation. The code of Murmur is available at https://github.com/uw-syfi/Murmur.

# MURMUR: An Efficient Inference System for Long-Form ASR 论文总结

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：长语音自动语音识别（Long-form ASR）需要在高精度和低延迟之间取得平衡。现有两类方法存在严重权衡：
  - **基于块的流水线**（如 WhisperX）：将音频分割成短窗口并行处理，延迟低，但丢失跨块上下文，导致说话人归属、时间戳对齐出现错误，需要脆弱的启发式方法拼接。
  - **长上下文模型**（如 VibeVoice-ASR）：一次推理处理整段音频，精度高，但推理延迟慢一个数量级（约10倍），且自回归解码容易陷入重复循环（repetition loops），导致整个请求失败。
- **整体含义**：MURMUR旨在打破这种权衡，在保持长上下文模型精度的同时，达到接近基于块流水线的低延迟，从而推动长语音ASR在实际场景中的高效部署。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
MURMUR在两级粒度上优化推理效率：
- **块间级（Inter-chunk）**：将块大小作为可调超参数，利用现代长上下文ASR模型无严格输入长度限制的特点，选择中间块大小（300秒）以平衡精度和延迟。
- **块内级（Intra-chunk）**：利用注意力稀疏性，对KV缓存实施滑动窗口驱逐策略，分别针对输出令牌和语音令牌进行优化。

### 关键技术细节
1. **块化推理**：
   - 使用VAD模型检测语音段边界，贪心合并为不超过300秒（c=300s）的块。
   - 多个块批量提交给推理引擎（vLLM或自定义后端），通过并行处理降低GPU空闲时间。

2. **KV缓存驱逐**：
   - **输出令牌驱逐**：基于StreamingLLM思想，保留固定数量的“注意力沉没”令牌（`n_sink`）和一个滑动窗口（`w`），驱逐中间令牌。公式：
     - 保留集 `K = {1,…,n_sink} ∪ {max(1, j-w+1), …, j}`
   - **语音令牌驱逐**：观察到语音令牌的注意力呈现对角稀疏模式（活跃区域随解码步数稳定前进）。引入延迟阶段（`d`步），固定窗口不移动，之后窗口随解码步数前进。公式：
     - 保留集 `R_j = {1,…,n_sink} ∪ {l_j, …, l_j + w}`，其中 `l_j = 0` for `j < d`，此后每步递增。
   - **理论加速比**：通过公式 `ρ = (Ls·T + T(T+1)/2) / ((n_sink + w)·T + T(T+1)/2)` 估算，300秒块下输出驱逐约1.51×，语音驱逐约1.06×（实际略低因管理开销）。

3. **输出拼接**：
   - 时间戳：将块内相对时间戳加上块偏移量得到绝对时间戳。
   - 说话人标签：利用模型内部说话人分配作为块内真值，仅在相邻块边界处使用外部说话人日志化结果计算置换映射 `π`，确保全局一致性，避免外部错误侵入模型已处理的区域。

## 3. 实验设计：数据集、基准、对比方法

## 6. 主要结论与发现

- **块大小是最关键的超参数**：300秒块匹配或超过单次推理精度（AMI-IHM上tcpWER 24.92% vs 25.68%），同时延迟降低3.68×（从370.7s降到100.8s）。更短块（30s）精度差，更长块（单次）失败风险高。
- **注意力高度稀疏**：在解码过程中，不到25%的语音令牌贡献99%的注意力权重，尤其在深层。这为驱逐提供了理论基础。
- **MURMUR达到最佳权衡**：
  - 无驱逐：3.68×加速，精度持平（tcpWER 24.92%）。
  - 输出驱逐：小幅提升（3.79×）但精度几乎不变。
  - 语音驱逐：4.20×加速，tcpWER 25.29%（仅退化<1%相对）。
  - 同时驱逐反而因管理开销略降（3.85×）。
- **鲁棒性提升**：单次推理在长/噪声录音上可能因重复循环完全失败，而MURMUR的块化将失败限制在单个块，整体更可靠。
- **最优上下文长度**：干净音频（Tedlium3、Earnings21）精度在较短块时饱和，噪声音频（AMI-SDM）需要更长块但也会在过长时下降，支持噪声积累假设。

## 7. 优点

- **方法创新**：
  - 首次系统分析长上下文ASR中块大小作为超参数的影响，发现中间值最优。
  - 提出针对语音令牌的延迟滑动窗口驱逐策略，利用了ASR解码特有的对角稀疏模式。
  - 输出拼接时巧妙保留模型内部说话人标签，只用外部diarization解决跨块身份，避免外部错误传播。
- **实验设计**：
  - 使用tcpWER作为主要指标，同时考核转录、说话人归属和时间戳，符合实际需求。
  - 考虑了失败情况，通过排除和单独报告两种方式展示真实部署风险。
  - 消融实验全面，揭示了驱逐策略的权衡。
- **可复现性**：开源代码（Apache 2.0），使用公开数据集和模型。

## 8. 不足与局限

- **模型局限性**：仅适用于能原生输出时间戳和说话人标签的长自回归ASR模型（当前仅有VibeVoice-ASR）。传统模型（如Whisper）需要外部后处理，无法在tcpWER上直接比较，通用性受限。
- **语言覆盖**：所有实验仅限英语。其他语言（不同韵律、形态、交谈模式）的行为未知。
- **驱逐策略非自适应**：采用固定的滑动窗口，虽然简单有效，但可能错过内容敏感的驱逐机会。论文指出内容感知的驱逐策略（如query-aware）是未来方向。
- **实验规模**：数据集较小（AMI仅8.7小时），统计显著性可能受限。Earnings21部分录音因长度超标被排除，影响代表性。
- **潜在风险**：自动转录系统可能被用于监控，需注意伦理部署。论文虽提及，但未讨论对策。
- **资源需求**：虽然延迟降低4.2倍，但绝对时间在长音频上仍需数十秒至上百秒，实时性仍有挑战。批大小为16时需要足够大的音频才能填满。
